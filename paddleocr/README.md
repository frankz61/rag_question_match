# PaddleOCR OCR 服务模块 —— 架构汇报

## 一、模块定位

本模块是整个题目匹配系统的**图像文字识别基础能力层**，承担将图片/PDF等非结构化输入转换为可处理文本的职责，为后续的语义检索、题目匹配等上层业务提供数据入口。

---

## 二、架构设计优势

### 1. 接口解耦，职责单一

服务以 **RESTful HTTP 接口**对外暴露能力，上层业务无需感知 OCR 引擎的实现细节。识别逻辑全部封装在 `ocr_match.py`，`main.py` 仅负责 HTTP 协议层，两层之间通过单一函数 `extract_text_from_image_bytes()` 交互，**高内聚、低耦合**。

### 2. 异步非阻塞，性能友好

FastAPI 本身是异步框架，而 PaddleOCR 的推理是 CPU/GPU 密集型的同步操作。本模块使用 `run_in_threadpool` 将推理任务投入线程池执行，**主事件循环不被阻塞**，确保服务在高并发场景下对其他请求保持响应能力。

### 3. 线程安全的 Reader 缓存机制

PaddleOCR 初始化成本极高（需加载模型权重），若每次请求都重新初始化则性能极差。本模块通过 `threading.local()` 实现**线程本地缓存**：

- 每个工作线程**首次使用时初始化一次**，后续请求复用，彻底消除重复加载开销
- 缓存以 `(use_angle_cls, lang, device)` 为键，支持多配置共存
- 线程隔离避免了多线程共享同一 reader 实例的竞态风险

### 4. 硬件自适应，部署灵活

运行时自动探测硬件环境并**优雅降级**：

- 配置 `gpu` 且检测到 CUDA → 使用 GPU 加速
- 配置 `gpu` 但无可用 CUDA → 自动降级为 CPU，服务不中断
- 配置 `auto` → 自动选择最优设备

同一套代码可以**无缝部署在开发机（CPU）和生产服务器（GPU）上**，无需修改任何代码。

### 5. 多格式支持，扩展性强

底层已具备处理 **PDF 和多种图片格式**（png/jpg/jpeg/bmp/tiff）的能力，PDF 接口可按需快速上线，扩展成本极低。

## 四、总结

本模块以**约 160 行代码**完成了一个生产就绪的 OCR 微服务的核心建设，在工程上重点解决了 OCR 引擎初始化开销大、同步阻塞、硬件环境差异等实际痛点，为整个题目匹配系统提供了稳定、高效的图像文字识别能力底座。后续只需打通 Milvus 检索链路，即可完成从"图片输入"到"题目匹配输出"的完整闭环。
